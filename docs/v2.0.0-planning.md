## Executive Summary

sailor v2.0.0 will be a **major version** release introducing **breaking changes** to improve API consistency, performance, and developer experience based on learnings from v1.0-v1.27.

**Goals**:
1. **API Modernization** — Unify inconsistent APIs across widgets and modules
2. **Performance** — Further optimize hot paths identified in production usage
3. **Ergonomics** — Simplify common patterns, reduce boilerplate
4. **Future-Proofing** — Prepare for Zig 0.16+ and ecosystem evolution

**Non-Goals**:
- Rewrite core architecture (v1.x architecture is sound)
- Remove major features (additive changes preferred)
- Break compatibility without migration path

---

## Breaking Changes

### 1. Buffer API Unification

**Problem**: Inconsistent cell manipulation methods (`setCell` vs `setChar`, `getCell` vs `getChar`)

**Change**:
```zig
// v1.x (inconsistent)
buffer.setChar(x, y, 'A', style); // Some widgets
buffer.setCell(x, y, cell);       // Other widgets (name collision)

// v2.0 (unified)
buffer.set(x, y, 'A', style);     // char + style
buffer.setCell(x, y, cell);       // full Cell struct
```

**Rationale**: Shorter, consistent naming. `set` is the default for character rendering.

**Migration**: Search-replace `setChar` → `set` (automated via script)

---

### 2. Widget Lifecycle Standardization

**Problem**: Some widgets use `init(allocator)`, others use `init(data)`, some have no `deinit`

**Change**:
```zig
// v1.x (inconsistent)
var tree = try Tree.init(allocator);  // allocator-based
var textarea = TextArea.init(&lines); // data-based
var list = List{ .items = &items };  // struct literal

// v2.0 (standardized)
var tree = try Tree.init(allocator, .{}); // allocator + options
var textarea = TextArea{ .lines = &lines }; // struct literal (no alloc)
var list = List{ .items = &items }; // same
```

**Rationale**: Clear distinction:
- **No allocator needed** → struct literal
- **Allocator needed** → `init(allocator, options)` + `deinit()`

**Migration**: Update widget construction calls (breaking change, requires code review)

---

### 3. Event System Overhaul

**Problem**: Event handling is callback-based with inconsistent signatures

**Change**:
```zig
// v1.x
terminal.draw(frame, |f| { ... }); // callback-based
terminal.pollEvent(timeout); // returns Event union

// v2.0 (proposed)
while (try terminal.nextEvent()) |event| {
    switch (event) {
        .key => |key_event| { ... },
        .mouse => |mouse_event| { ... },
        .resize => |new_size| { ... },
    }
}
```

**Rationale**: Iterator-style API is more idiomatic in Zig

**Migration**: Rewrite event loops (significant breaking change)

---

### 4. Style API Simplification

**Problem**: Color specification is verbose (`Color{ .basic = .red }`)

**Change**:
```zig
// v1.x
.style = Style{ .fg = Color{ .basic = .red }, .bg = Color{ .indexed = 235 } }

// v2.0
.style = Style{ .fg = .red, .bg = .@"235" } // color inference
```

**Rationale**: 90% of use cases are basic colors. Advanced colors can use explicit constructors.

**Migration**: Search-replace patterns (mostly automated)

---

### 5. Remove Deprecated APIs

**Deprecations to remove** (introduced in v1.x):
- `Rect.new()` → Use `Rect{ .x = ..., .y = ..., .width = ..., .height = ... }`
- `Block.withTitle()` → Use `.title` field directly
- Old constraint syntax (`Constraint.Length(10)`) → Use `.{ .length = 10 }`

**Rationale**: Clean up API surface, reduce maintenance burden

**Migration**: Warnings emitted in v1.27, remove in v2.0

## New Features

### 1. Async Event Loop (Optional)

**Motivation**: Enable non-blocking I/O for network widgets (HTTP, WebSocket, SSH)

**Proposal**:
```zig
var terminal = try AsyncTerminal.init(allocator);
defer terminal.deinit();

while (try terminal.poll()) |event| {
    // handle events
}

// Widgets can schedule async tasks
try http_widget.scheduleRequest("https://api.example.com");
```

**Impact**: Additive change, opt-in for apps needing async

---

### 2. Theming V2 (Semantic Colors)

**Motivation**: Current themes are palette-based, not semantic

**Proposal**:
```zig
// v1.x
.style = Style{ .fg = .blue }; // hard-coded color

// v2.0
.style = Style{ .fg = .@"ui.primary" }; // semantic color from theme
```

**Benefits**:
- Themes can define semantic meanings (primary, secondary, error, warning)
- Dark/light mode switching without code changes
- Better accessibility (colorblind themes)

**Impact**: Additive, backward compatible if `.fg = .red` still works

---

### 3. GPU-Accelerated Rendering (Experimental)

**Motivation**: TUI performance is CPU-bound for complex UIs

**Proposal**: Use [notcurses](https://github.com/dankamongmen/notcurses) or similar for hardware-accelerated rendering

**Status**: Research phase, may defer to v2.1+ if complexity is high

---

### 4. Web Assembly Target

**Motivation**: Run sailor apps in browsers via wasm + WebGL terminal

**Proposal**: Add `wasm32-emscripten` target support

**Status**: Proof-of-concept needed, may defer to v2.1+

## Migration Guide

### Automated Migration Script

Provide `scripts/migrate-v1-to-v2.zig`:
```bash
zig run scripts/migrate-v1-to-v2.zig -- /path/to/project
```

**Automated replacements**:
- `setChar` → `set`
- `Color{ .basic = .X }` → `.X`
- `Rect.new(...)` → `Rect{ ... }`

**Manual review needed**:
- Widget lifecycle changes (init/deinit)
- Event loop rewrites
- Semantic color adoption (opt-in)

---

### Deprecation Timeline

**v1.27** (current):
- Deprecation warnings for APIs to be removed in v2.0
- New APIs introduced alongside old ones

**v1.28** (next):
- All v2.0 APIs available in parallel
- Consumer projects can test migration incrementally

**v2.0** (major):
- Old APIs removed
- Breaking changes enforced
- Migration script provided

**v2.1+** (post-major):
- Additive changes only (no more breaking changes until v3.0)
